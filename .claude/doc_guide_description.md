# Doc-update directive

You are updating the hand-written portions of the snapcap SDK documentation
site. This directive is read by `scripts/update-docs.sh` and passed to
`claude -p` as the prompt.

## Project context

`@snapcap/native` is a Node SDK that talks to `web.snapchat.com` natively
— no Playwright, no headless Chrome. It runs Snap's web JS bundle inside
a `vm.Context` sandbox with browser-API shims, drives gRPC-Web over the
host's fetch, and exposes a manager-class API surface (Friends, Messaging,
Inbox, Media, Presence, Stories) to consumers.

The full architecture is described in `CLAUDE.md` at the repo root.

## Voice and content rules

- **Consumer-facing only.** No internal-dev rationale, no Phase notes,
  no references to memory files (e.g. `feedback_*.md`), no project
  history, no migration status.
- **Direct, technical, honest.** Match the voice of the existing TypeDoc
  output: brief one-line summary, then supporting detail.
- **Show, don't tell.** Every guide page should have at least one runnable
  code example consumers can copy-paste.
- **Cross-link to the API reference.** Use Fumadocs MDX link syntax to
  reference auto-generated pages: `[SnapcapClient](/docs/api/classes/SnapcapClient)`.
- **No fictional API.** If you're tempted to write a code example showing
  `client.foo.bar()` and you haven't verified that method exists, DON'T.

## File format requirements (Fumadocs)

Every new `.mdx` file MUST start with YAML frontmatter:

```mdx
---
title: Page Title Here
description: One-sentence description for the sidebar / search.
---

# Page Title Here

Body content here...
```

When creating a new page in a directory, also update the directory's
`meta.json` to register the page in the sidebar. Inspect the existing
file for the real shape — it lists the actual page basenames in display
order. Example (verify the current contents before editing):

```json
{
  "title": "Guide",
  "pages": ["auth", "persistence", "logging", "throttle", "multi-tenant"]
}
```

Order pages by reader journey: introductory pages first, advanced topics
later. Group related pages near each other.

### Fumadocs MDX components available inline

Fumadocs ships several React components that work directly in MDX. Use
them when they materially help readability — don't sprinkle them for
decoration. Cheap quick reference (full list at https://fumadocs.dev):

```mdx
import { Callout } from "fumadocs-ui/components/callout";
import { Tab, Tabs } from "fumadocs-ui/components/tabs";
import { Card, Cards } from "fumadocs-ui/components/card";

<Callout type="warn">
  Required: `userAgent` is no longer optional in `BrowserContext`.
</Callout>

<Tabs items={["per-instance", "shared"]}>
  <Tab value="per-instance">...</Tab>
  <Tab value="shared">...</Tab>
</Tabs>
```

Imports go INSIDE the MDX body (under the frontmatter, before the first
heading) — Fumadocs hoists them. Don't import a component you don't
actually use; lint will warn.

## What "meaningful" change means

UPDATE / CREATE hand-written docs for:
- New public types or methods on `SnapcapClient` or any manager
- New constructor options
- New public exports from `src/index.ts`
- Behavior changes that affect how consumers use the SDK
- New patterns or idioms (e.g. shared throttle, multi-tenant config)

SKIP hand-written doc updates for:
- Pure internal refactors (bundle/, shims/, transport/)
- Comment-only changes
- Test-only changes
- TypeDoc regeneration (handled separately by `bun run docs:api`)
- Dependency bumps, lockfile changes, gitignore updates
