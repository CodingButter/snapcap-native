The persistence layer — the `DataStore` interface, built-in implementations, the Web Storage shim, the cookie store, and IDB helpers. See [Persistence](/docs/guide/persistence) for the conceptual overview and [internals/io-overrides](/docs/internals/io-overrides) for why each storage area is overridden.

## DataStore

```ts
interface DataStore {
  get(key: string): Promise<Uint8Array | undefined>;
  set(key: string, value: Uint8Array): Promise<void>;
  delete(key: string): Promise<void>;
}
```

The single abstraction the entire SDK uses to persist state. Implement these three async methods and you've got a working backend (Redis, KMS, Postgres, IndexedDB, anything).

For Web Storage shim support (`localStorage` / `sessionStorage` over the same DataStore) and synchronous `document.cookie` reads, the SDK additionally consults synchronous helpers if present:

```ts
// Optional — without these, StorageShim falls back to a startup-loaded cache.
getSync(key: string): Uint8Array | undefined;
setSync(key: string, value: Uint8Array): void;
keys(prefix?: string): string[];
```

The two built-in stores (`FileDataStore`, `MemoryDataStore`) implement all three.

## FileDataStore

```ts
class FileDataStore implements DataStore {
  constructor(filePath: string);
  get(key: string): Promise<Uint8Array | undefined>;
  set(key: string, value: Uint8Array): Promise<void>;
  delete(key: string): Promise<void>;
  getSync(key: string): Uint8Array | undefined;
  setSync(key: string, value: Uint8Array): void;
  keys(prefix?: string): string[];
}
```

Single JSON file on disk, in-memory cache loaded at construction, eager flush on every write. The recommended default for non-production use.

```ts
import { FileDataStore } from "@snapcap/native";
const store = new FileDataStore("./auth/perdyjamie.json");
```

The on-disk shape is `Record<string, number[]>` (each value is `Array.from(Uint8Array)`). Corrupt file → starts fresh on next boot. The directory is created automatically on first write.

## MemoryDataStore

```ts
class MemoryDataStore implements DataStore {
  constructor();
  get(key: string): Promise<Uint8Array | undefined>;
  set(key: string, value: Uint8Array): Promise<void>;
  delete(key: string): Promise<void>;
  getSync(key: string): Uint8Array | undefined;
  setSync(key: string, value: Uint8Array): void;
  keys(prefix?: string): string[];
}
```

In-memory only. Useful for tests, ephemeral one-shot jobs. Loses everything on process exit — every boot is a cold start.

```ts
import { MemoryDataStore } from "@snapcap/native";
const store = new MemoryDataStore();
```

## StorageShim

```ts
class StorageShim implements Storage {
  constructor(store: DataStore, prefix: string);
}
```

A Web Storage API shim (`getItem` / `setItem` / `removeItem` / `clear` / `key` / `length`) backed by a `DataStore`. Used internally by the `LocalStorageShim` / `SessionStorageShim` sandbox shims to back the sandbox's `localStorage` (`prefix: "local_"`) and `sessionStorage` (`prefix: "session_"`). Each prefix namespace keeps multiple shims collision-free on the same DataStore.

If the underlying store implements the optional sync helpers, the shim hits them directly. Otherwise it falls back to a startup-populated cache (best-effort sync read).

You'll rarely need to instantiate this directly — `Sandbox` does it for you when you pass `dataStore` in `SandboxOpts`. Surfaced for advanced consumers that want to share a DataStore namespace with their own app code.

## CookieJarStore

```ts
class CookieJarStore {
  readonly jar: CookieJar;
  static create(store: DataStore, key?: string): Promise<CookieJarStore>;
  flush(): Promise<void>;
}
```

A `tough-cookie` `CookieJar` whose serialised state lives under a single DataStore key (default `"cookie_jar"`). Loads at construction; persist with `flush()`.

```ts
import { CookieJarStore, FileDataStore } from "@snapcap/native";
const store = new FileDataStore("./auth.json");
const cs = await CookieJarStore.create(store);
await someCookieAwareFetch(cs.jar, ...);
await cs.flush();   // persist any changes
```

The SDK uses this internally to back `client.makeRpc()` and `transport/cookies.ts`. Lets `fetch()` wrappers attach cookies automatically while everything else (bearer, bundle storage) shares the same backing DataStore.

The same jar is also wired into the sandbox via `CookieContainerShim` (happy-dom outgoing fetch), `DocumentCookieShim` (JS-level reads/writes), and `WebSocketShim` (upgrade GET cookies). All paths see each other's writes — see [internals/shims](/docs/internals/shims).

## IDB helpers

Promise-friendly IndexedDB helpers that go through the sandbox's `indexedDB` global. With a DataStore configured, writes land at `indexdb_<db>__<store>__<key>`.

### idbGet

```ts
function idbGet<T = unknown>(dbName: string, storeName: string, key: string): Promise<T | undefined>
```

Read a JSON-decoded value. `undefined` if absent.

### idbPut

```ts
function idbPut(dbName: string, storeName: string, key: string, value: unknown): Promise<void>
```

Write `value` (JSON-encoded) at the given coordinates. Creates the object store on first use.

### idbDelete

```ts
function idbDelete(dbName: string, storeName: string, key: string): Promise<void>
```

Delete the entry. No-op if absent.

```ts
import { idbGet, idbPut, idbDelete } from "@snapcap/native";

await idbPut("myapp", "settings", "theme", { mode: "dark" });
const settings = await idbGet<{ mode: string }>("myapp", "settings", "theme");
await idbDelete("myapp", "settings", "theme");
```

Each call opens the database, runs one op, and waits for the transaction to settle. The shim is cheap so we don't bother caching the `IDBDatabase` across calls.
